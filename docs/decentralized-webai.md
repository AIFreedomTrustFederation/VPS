# Decentralized WebAI

WebAI should be a decentralized, open-source, shared-intelligence layer for AIFT VPS.

It should not be a premium model where one user has a stronger assistant than another user because of private access to a larger closed model.

## Goal

The goal is one shared WebAI network that learns, improves, and serves users through the whole AIFT VPS decentralized compute network.

```text
AIFT VPS provider nodes
  -> open-source model runtimes
  -> shared WebAI services
  -> no-code app building
  -> general-purpose user assistance
```

## Equality principle

Every user should interact with WebAI as part of the same shared network intelligence.

The network may route requests to different nodes based on availability, device type, model size, and workload need, but the product should not intentionally create a permanent intelligence gap where one user receives a better class of assistant than another user.

## Practical fairness

Equal access does not mean every device runs the same model locally.

A phone may run a smaller local helper. A server may run a larger model. A cluster may handle a complex job. The fairness goal is that the user gets access to the best available shared network capability, not only the capability of their own device.

## Shared learning model

WebAI should improve through network-level learning patterns such as:

- shared app templates
- shared build solutions
- shared documentation improvements
- shared error fixes
- shared routing patterns
- shared user intent patterns
- shared reusable components

Personal private data should not be treated as public training data by default.

## General-purpose assistant

WebAI should not be limited to VPS administration.

Users should be able to talk about anything needed to create an app, including:

- business ideas
- website copy
- app features
- databases
- user flows
- design direction
- forms
- automations
- emails
- marketing pages
- dashboards
- APIs
- data models
- no-code logic
- deployment plans

The answer can then become an app, template, workflow, message, page, API, database schema, or deployment plan.

## No-code app building

WebAI should help users create apps by conversation.

The no-code flow should be:

```text
User describes idea
WebAI asks clarifying questions
WebAI creates project plan
WebAI selects template
WebAI creates pages and components
WebAI connects data sources
WebAI prepares build settings
WebAI assigns eligible nodes
WebAI shows preview
User approves release
WebAI records disclosure
```

## Network-based compute

WebAI inference and app-building work should use the decentralized provider network where possible.

Suggested work placement:

```text
small chat turns -> local or nearby small model
repo summary -> laptop, desktop, or server node
build planning -> laptop, desktop, or server node
large reasoning task -> eligible model node or cluster
app build -> build-capable provider node
disclosure and audit -> control-plane or verified node
```

## Real-data rule

WebAI should use real data from:

- AIFT VPS repository
- authorized connected repositories
- local registries
- local heartbeat files
- local node cards
- local logs
- real build records
- real release records
- real disclosure records

If data is missing, WebAI should say what is missing and guide the user to connect or create it.

## User privacy rule

The shared network should not expose private user conversations, private repository contents, private app ideas, or private business data to other users.

Shared learning should focus on reusable patterns, not private raw user data.

## Long-term vision

WebAI becomes the conversational operating system for AIFT VPS:

```text
Talk to WebAI
Build any app
Run it on decentralized compute
Know where it runs
Keep everything open-source
Let the network improve together
```
